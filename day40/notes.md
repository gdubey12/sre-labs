# Day 40 — Helm Capstone: Multi-Tier App With a Hand-Written Chart

## Goal

Combine everything from Days 36–39 into one realistic chart: a multi-tier app
(frontend → backend → Redis cache) built entirely by hand, deployed as a
single Helm release.

```
Frontend (nginx, reverse-proxy via custom config)
        ↓ proxy_pass via Service DNS
Backend (httpbin)
        ↓ (wiring proven, not yet consumed by app code)
Redis (subchart dependency, standalone, no persistence)
```

---

## Chart Structure

```
multitier/
├── Chart.yaml              # declares redis as a dependency
├── Chart.lock               # generated by helm dependency update
├── values.yaml              # frontend.*, backend.*, redis.* namespaces
├── charts/
│   └── redis-27.0.4.tgz     # vendored dependency
└── templates/
    ├── _helpers.tpl
    ├── frontend-deployment.yaml
    ├── frontend-service.yaml
    ├── frontend-configmap.yaml   # nginx reverse-proxy config
    ├── backend-deployment.yaml
    └── backend-service.yaml
```

## Chart.yaml — Dependency Declared Upfront

```yaml
apiVersion: v2
name: multitier
description: A multi-tier app - frontend (nginx) + backend (httpbin) + Redis cache
type: application
version: 0.1.0
appVersion: "1.0.0"

dependencies:
  - name: redis
    version: "27.0.4"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled
```

Same pattern as Day 38, just declared from the start instead of added later.

## values.yaml — Per-Tier Namespacing

```yaml
frontend:
  replicaCount: 1
  image:
    repository: nginx
    tag: "1.25.0"
    pullPolicy: IfNotPresent
  service:
    type: NodePort
    port: 80

backend:
  replicaCount: 1
  image:
    repository: kennethreitz/httpbin
    tag: "latest"
    pullPolicy: IfNotPresent
  service:
    type: ClusterIP
    port: 80

redis:
  enabled: true
  architecture: standalone
  auth:
    enabled: false
  master:
    persistence:
      enabled: false   # avoids the Day 38 storage-class-missing issue proactively

nameOverride: ""
fullnameOverride: ""
```

Design pattern: `frontend.*` and `backend.*` are top-level namespaces invented
for this chart — same convention as `redis.*` for the subchart, just applied
to your own components. This is how multi-component charts stay organized.

## _helpers.tpl — Why No Shared selectorLabels This Time

Day 39's chart had ONE component, so one shared `selectorLabels` helper made
sense. Today's chart has TWO independent tiers (frontend, backend) plus a
Redis subchart — if frontend and backend shared the same selector labels,
their Services couldn't tell each other's pods apart:

```
Frontend Service selector: app.kubernetes.io/name: multitier
Backend Service selector:  app.kubernetes.io/name: multitier   ← SAME, BROKEN

Result: Services can't distinguish frontend pods from backend pods.
Traffic could route to the wrong tier.
```

### Design decision made
- `multitier.labels` helper only holds chart-wide identity (chart name+version,
  app version, managed-by) — no `app.kubernetes.io/name`/`instance`
- Each tier (frontend, backend) defines its own
  `app.kubernetes.io/component: frontend|backend` label inline, directly in
  its own Deployment + Service templates
- This label must match in exactly 3 places per tier: Deployment pod labels,
  Deployment selector, Service selector

```yaml
{{- define "multitier.labels" -}}
helm.sh/chart: {{ printf "%s-%s" .Chart.Name .Chart.Version }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}
```

No "right" universal answer here — Helm doesn't enforce this. The chart
author decides based on how many independently-selectable components the
chart has.

## Backend Templates

### backend-deployment.yaml
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "multitier.fullname" . }}-backend
  labels:
    {{- include "multitier.labels" . | nindent 4 }}
    app.kubernetes.io/component: backend
spec:
  replicas: {{ .Values.backend.replicaCount }}
  selector:
    matchLabels:
      app.kubernetes.io/name: {{ include "multitier.name" . }}
      app.kubernetes.io/instance: {{ .Release.Name }}
      app.kubernetes.io/component: backend
  template:
    metadata:
      labels:
        {{- include "multitier.labels" . | nindent 8 }}
        app.kubernetes.io/name: {{ include "multitier.name" . }}
        app.kubernetes.io/instance: {{ .Release.Name }}
        app.kubernetes.io/component: backend
    spec:
      containers:
        - name: backend
          image: "{{ .Values.backend.image.repository }}:{{ .Values.backend.image.tag }}"
          imagePullPolicy: {{ .Values.backend.image.pullPolicy }}
          ports:
            - name: http
              containerPort: 80
              protocol: TCP
```

### backend-service.yaml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: {{ include "multitier.fullname" . }}-backend
  labels:
    {{- include "multitier.labels" . | nindent 4 }}
    app.kubernetes.io/component: backend
spec:
  type: {{ .Values.backend.service.type }}
  ports:
    - port: {{ .Values.backend.service.port }}
      targetPort: http
      protocol: TCP
      name: http
  selector:
    app.kubernetes.io/name: {{ include "multitier.name" . }}
    app.kubernetes.io/instance: {{ .Release.Name }}
    app.kubernetes.io/component: backend
```

`ClusterIP` deliberately — backend should never be reachable from outside the
cluster directly, only via the frontend's proxy.

## Frontend Templates — Cross-Tier Communication via Service DNS

### frontend-configmap.yaml — the key pattern of the day

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "multitier.fullname" . }}-frontend-conf
  labels:
    {{- include "multitier.labels" . | nindent 4 }}
    app.kubernetes.io/component: frontend
data:
  default.conf: |
    server {
      listen 80;

      location / {
        return 200 'Frontend is up. Try /api/ to reach the backend.\n';
        add_header Content-Type text/plain;
      }

      location /api/ {
        proxy_pass http://{{ include "multitier.fullname" . }}-backend.{{ .Release.Namespace }}.svc.cluster.local/;
      }
    }
```

**The critical line:**
```
proxy_pass http://{{ include "multitier.fullname" . }}-backend.{{ .Release.Namespace }}.svc.cluster.local/;
```
Builds the full Kubernetes internal DNS name:
`<servicename>.<namespace>.svc.cluster.local`

This is how one tier finds another in a multi-tier chart — always by Service
DNS, never by pod IP (pod IPs are ephemeral and change on every restart).

### frontend-deployment.yaml — mounting a ConfigMap as a file

```yaml
volumeMounts:
  - name: nginx-conf
    mountPath: /etc/nginx/conf.d/default.conf
    subPath: default.conf
volumes:
  - name: nginx-conf
    configMap:
      name: {{ include "multitier.fullname" . }}-frontend-conf
```

**New pattern vs Day 39:** Day 39 used `envFrom` + `configMapRef` to inject
ConfigMap data as environment variables. Here, the ConfigMap is mounted **as a
file** instead — necessary because nginx reads its config from a file, not
env vars.

`subPath: default.conf` is essential — without it, the ENTIRE ConfigMap would
mount as a directory at `/etc/nginx/conf.d/default.conf`, which is wrong (that
path expects a file, not a directory) and would wipe out anything else
normally in `/etc/nginx/conf.d/`. `subPath` extracts just that one key from
the ConfigMap and mounts it as a single file.

### frontend-service.yaml
```yaml
spec:
  type: {{ .Values.frontend.service.type }}   # NodePort — external access point
```

`NodePort` deliberately, contrasted with backend's `ClusterIP` — frontend is
the only tier meant to be reached from outside the cluster.

## Installation — Avoiding the Day 38 Bitnami Index Bug Proactively

Lesson carried forward from Day 38: never install a chart-with-dependency
directly from a directory if the dependency repo (e.g. Bitnami) has an
oversized index. Package first, then install from the `.tgz`:

```bash
helm dependency update     # pulls redis-27.0.4.tgz into charts/
helm template multitier .  # render-check locally before installing
helm package .              # bundles chart + charts/ into multitier-0.1.0.tgz
helm install multitier multitier-0.1.0.tgz --namespace day40-demo --create-namespace
```

### Why packaging avoids the issue
| Command | What it reads |
|---------|---------------|
| `helm install multitier .` | Directory + may re-check live repo indexes for dependencies |
| `helm install multitier multitier-0.1.0.tgz` | Only the self-contained archive — fully offline-safe once packaged |

This is also how chart distribution actually works in production —
`helm install bitnami/nginx` installs a `.tgz` Helm fetched from the repo, not
a raw directory.

## Verification — Proving Cross-Tier Wiring End to End

```bash
kubectl get all -n day40-demo
```
Result: all three tiers `1/1 Running` on first deploy — frontend Deployment,
backend Deployment, Redis StatefulSet, plus their Services.

### Testing the frontend directly
```bash
curl http://<worker-ip>:<nodeport>/
# → "Frontend is up. Try /api/ to reach the backend."
```

### Testing the proxy to backend (the real proof)
```bash
curl http://<worker-ip>:<nodeport>/api/get
```
Result — httpbin's response revealed the actual internal request path:
```json
{
  "headers": { "Host": "multitier-backend.day40-demo.svc.cluster.local" },
  "url": "http://multitier-backend.day40-demo.svc.cluster.local/get"
}
```
This confirms the full chain:
```
curl → NodePort → frontend Service → frontend pod (nginx)
    → proxy_pass → multitier-backend.day40-demo.svc.cluster.local
    → backend Service → backend pod (httpbin)
    → response flows back through the same path
```

**Note on trailing slash:** `/api` (no trailing slash) returned a 301 redirect
— nginx's `location /api/` block requires the exact trailing slash to match.
Always test with the trailing slash, or use `curl -L` to follow redirects.

### Redis health check
```bash
kubectl exec -n day40-demo multitier-redis-master-0 -- redis-cli ping
# → PONG
```

## Honest Note: Redis's Role in This Lab

Redis is deployed, healthy, and reachable via Service DNS — but nothing in
the frontend or backend currently calls it. `PONG` only proves the Redis pod
itself is alive.

**What this exercise actually proved:**
```
1. Can you declare a real subchart dependency in Chart.yaml?          ✅
2. Can you configure it via your own values.yaml?                     ✅
3. Does it deploy alongside custom templates in one release?          ✅
4. Is it healthy and reachable on its Service DNS?                    ✅
```

This mirrors a common real-world pattern: scaffold the architecture and wire
up infrastructure dependencies before application code is written to
actually consume them.

**What real usage would look like** (for future reference, if backend code
is ever owned/written):
```python
import redis
r = redis.Redis(host='multitier-redis-master.day40-demo.svc.cluster.local', port=6379)
r.set('key', 'value')
```
Same Service-DNS pattern as the frontend→backend proxy — just from
application code instead of an nginx config.
